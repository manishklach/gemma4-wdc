Architecture

Shared Execution Runtime is a runtime layer, not another agent shell.

Its job is to catch semantic overlap before backend work begins, collapse matching tasks into a shared execution unit, and make the whole decision path observable.

Core components

  • Task ingestion API: accepts typed tasks with branch and resource metadata.
  • Task fingerprinting: builds an exact structural hash and semantic comparison text.
  • Admission window: bounded and non-resetting, started by the first arrival.
  • Shared execution unit: owns subscribers, execution state, and result fan-out.
  • Observability: exposes tasks received, SEUs created, executions saved, and false-collapse behavior.

Data flow

  1. Tasks arrive from concurrent branches.
  2. The runtime computes a task fingerprint and resource scope.
  3. Pending SEUs are checked for exact structural matches first.
  4. Semantic overlap is only considered after exact matching fails.
  5. Matching tasks become subscribers during the admission window.
  6. When the window expires, the SEU executes once and fans the result out.

Why this is credible for coding agents

Repo-understanding branches often repeat the same structural work with slightly different phrasing. This runtime is aimed squarely at that class of workload: repo scan, code search, dependency tracing, and test planning across neighboring branches.

Prototype limits

The current implementation is in-memory and local. It demonstrates the mechanics clearly, but it does not yet solve distributed coordination, persistence, or live production tool integration.
