Architecture

Gemma4-WDC is a middleware layer for local agents, not another agent shell.

Its job is to catch overlap before backend work begins, group compatible tasks into shared execution units, and make the whole decision path inspectable on a laptop.

Core components

  • Agent task generation: simulated or hybrid local agents emit structured tool tasks.
  • Task ingress registry: records agent, branch, task type, resource hint, and arrival time.
  • Fingerprinting: computes exact structural hashes and lightweight semantic comparison text.
  • Admission control: opens a bounded, non-resetting window on first arrival.
  • SEU manager: owns subscriber state, execution lifecycle, and result fan-out.
  • Observability: exposes counts, savings, and safety behavior.

Why this architecture is laptop-first

The architecture assumes one machine, one central FastAPI service, lightweight agents, and mock or lightweight local tool backends. Hybrid mode adds one optional real model adapter, but the design does not depend on many concurrent heavy local models.

Coding-agent focus

Repo-understanding branches are a practical place to prove the systems thesis. Several branches can ask where a state transition happens, what a module does, or where the benchmark harness lives. The middleware turns that into shared backend work instead of repeated scans.

Prototype limits

The current system is local and in-memory. It demonstrates task overlap, bounded admission, shared execution, and fan-out clearly, but it does not yet solve distributed coordination, persistence, or a strong production-grade real-model integration path.
